# Cup Stacking Evaluation Alignment

The cup-stacking generation and reference evaluation environments load the
same kitchen scene and table asset. The visible mismatch comes from the cup
start-position distribution, not from a different table size.

| Object | Evaluation x range | Evaluation y range |
|---|---:|---:|
| `blue_cup` | `0.31` to `0.41` | `-0.45` to `-0.35` |
| `pink_cup` | `0.41` to `0.51` | `-0.45` to `-0.35` |

The original generation defaults use a different workspace and may also flip
the blue/pink relative layout. The provided evaluation program is only a
reference, so use its distribution as the center of a slightly wider training
distribution rather than assuming leaderboard parameters are fixed.

## Generate Evaluation-Aligned Data

Run inside the Isaac Lab Docker container. Use a new dataset repo instead of
mixing compact demonstrations into an older dataset.

```bash
DATASET_REPO=MidoriChou/ai_final_9_compact_wide

env -u DISPLAY \
VK_ICD_FILENAMES=/usr/share/vulkan/icd.d/nvidia_icd.json \
python scripts/datagen/generate.py \
  --task HCIS-CupStacking-SingleArm-v0 \
  --num_envs 1 \
  --device cuda \
  --headless \
  --enable_cameras \
  --record \
  --use_lerobot_recorder \
  --lerobot_dataset_repo_id "$DATASET_REPO" \
  --object_poses data/ai_final_7/object_poses.json \
  --cup_layout_profile eval_wide \
  --target_demo_count 1000
```

`--cup_layout_profile eval` samples the two cup positions independently from
the same initial positions and `+/- 5 cm` x/y offsets used by the reference
evaluation. It intentionally ignores the source cup positions, workspace
clamp, pair flipping, and minimum/maximum cup-distance settings.

`eval_wide` expands every evaluation x/y edge by 2 cm while staying inside the
scripted Franka policy's reliable workspace:

| Object | Wide x range | Wide y range |
|---|---:|---:|
| `blue_cup` | `0.29` to `0.43` | `-0.47` to `-0.33` |
| `pink_cup` | `0.39` to `0.53` | `-0.47` to `-0.33` |

Only successful state-machine episodes are exported. Compact demonstrations
contain about 600 frames instead of the old roughly 1200 frames, so 1000
compact demonstrations have about the same frame count as 500 old
demonstrations while covering twice as many layouts. The generator flushes the
final successful episode before exiting, so `--target_demo_count 1000`
produces 1000 exported episodes rather than 999.

If generation is interrupted after at least one episode was exported, rerun
the same command with `--resume`. The generator reads the existing episode
count from the Hugging Face LeRobot cache and continues until the total reaches
`--target_demo_count`:

```bash
--resume \
--target_demo_count 1000
```

Use normal `Ctrl+C` cleanup when possible. A forced kill can leave metadata
incomplete; the generator detects that case before creating the environment
and requires a clean dataset directory instead of attempting an unsafe resume.

Upload the completed dataset from inside the container:

```bash
hf upload MidoriChou/ai_final_9_compact_wide \
  ~/.cache/huggingface/lerobot/MidoriChou/ai_final_9_compact_wide/ \
  --repo-type dataset
```

## Train Candidate Models

Run training on the host. Do not package the model from the lowest training
loss alone; train candidates, evaluate them under the same representative
settings and seeds, then package the one with the highest success count.

ACT is the recommended first candidate for this dataset because the
demonstrations are consistent, single-task, and generated by one scripted
trajectory family. The training helper changes ACT from its default 100
open-loop actions to per-step replanning. Temporal ensembling is disabled by
default because averaging the discrete gripper `open` and `close` commands can
delay the grasp and make the robot close on empty space.

```bash
scripts/train_cup_stacking_policy.sh \
  act \
  MidoriChou/ai_final_9_compact_wide \
  outputs/train/cup_compact_wide_act_seed42
```

Use Diffusion as the second comparison after the aligned dataset is working.
The helper uses a smaller network suitable for an 8 GB GPU and DDIM inference
to avoid excessively slow evaluation.

```bash
scripts/train_cup_stacking_policy.sh \
  diffusion \
  MidoriChou/ai_final_9_compact_wide \
  outputs/train/cup_compact_wide_diffusion_seed42
```

The helper defaults to 200,000 steps, batch size 4, AMP, moderate image
augmentation, and checkpoints every 20,000 steps. Override settings with
environment variables:

```bash
BATCH_SIZE=2 STEPS=300000 WANDB_ENABLE=true \
  scripts/train_cup_stacking_policy.sh act MidoriChou/ai_final_9_compact_wide
```

The compact cup-stacking state machine records 605-step demonstrations instead
of the previous roughly 1200-step demonstrations. It reaches the hand-origin
grasp height while the gripper is open, then closes while holding that exact
pose. Do not mix old slow demonstrations with compact demonstrations in one
dataset; create a new dataset repo and retrain.

To test an older ACT checkpoint without temporal ensembling:

```bash
python scripts/rollout.py \
  --task=eval/cup_stacking_eval.py \
  --policy_type=lerobot-act \
  --policy_checkpoint_path=<checkpoint/pretrained_model> \
  --policy_action_horizon=1 \
  --disable_act_temporal_ensemble \
  --device=cuda \
  --enable_cameras \
  --eval_rounds=5 \
  --episode_length_s=20
```

## Select the Best Checkpoint

The provided evaluation command is a reference, not a guaranteed leaderboard
configuration. Keep the checkpoint self-contained and test each candidate with
multiple seeds. Use the same local settings for every candidate so their
success counts are comparable:

```bash
python scripts/rollout.py \
  --task=eval/cup_stacking_eval.py \
  --policy_type=lerobot-act \
  --policy_checkpoint_path=outputs/train/cup_compact_wide_act_seed42/checkpoints/last/pretrained_model \
  --policy_action_horizon=1 \
  --device=cuda \
  --enable_cameras \
  --seed=42 \
  --eval_rounds=50 \
  --episode_length_s=60
```

Evaluate the last checkpoint and earlier saved checkpoints. Training loss can
continue decreasing after simulation success has peaked. Compare at least
`100000`, `140000`, `180000`, and `last`, using the same 50 evaluation rounds
for every candidate. Repeat the comparison with several seeds. The generated
data and checkpoint config should carry the behavior; do not rely on a
particular evaluator timeout or action-horizon argument.

## Create the Submission Zip

After choosing the best evaluated checkpoint:

```bash
python scripts/package_policy.py \
  outputs/train/cup_compact_wide_act_seed42/checkpoints/180000/pretrained_model \
  --reactive-act
```

This validates the model, config, preprocessing, postprocessing, and
normalization files. For ACT, `--reactive-act` also stores per-step replanning
and disables temporal action averaging inside the submitted checkpoint, so the
behavior does not depend on evaluator-side overrides. It then creates:

```text
checkpoints/my_policy/
checkpoints/my_policy.zip
```

Upload `checkpoints/my_policy.zip`. The zip contains `my_policy/` as its root
directory and includes all files required for inference.
